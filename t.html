Why

The Escalations tab in each service module holds a single table that is closely related to the work shown in the Requests tab. Splitting it into its own tab forces an extra click and fragments the request/escalation picture. Consolidating the escalations table at the bottom of the Requests view gives a single scrollable place to see open work, closed work, and escalations together.

What

For each module that has an Escalations tab, move the existing Escalations table to the bottom of its main Requests-style tab — below the last existing table — and remove the standalone "Escalations" tab button. The table itself (columns, sorting, pagination, data source) does not change; only its location in the page does. The "Escalations" section title and its descriptive subtitle are kept so the table stays distinguishable from the request tables.

Modules in scope: Managed HR, Managed Payroll, Talent Acquisition. EDC is explicitly out of scope — it keeps its Escalations tab (it has no Requests/Closed table to anchor the move).

Done when: none of the three pages show an "Escalations" tab button; each shows its Escalations table appended below the last table of its main tab; sorting/pagination still work; npx tsc --noEmit is clean.

Patterns To Follow

  • Each module already stacks tables inside one tab by wrapping each in a spacing <div className="mt-8"> (see Payroll Closed Requests at PayrollPage.tsx:1244). Reuse that exact spacing wrapper for the moved Escalations section.
  • The escalations data (escalations, sortedEscalations, escLoading, escPage, escTotalPages, handleEscSort) is loaded in a mount/clientId-scoped effect that is not gated on activeTab (confirmed at HRManagedPage.tsx:630–644). Moving the table does not require touching any fetch logic.
  • The Escalations block is fully self-contained inside a <CollapsibleSection> — it can be relocated verbatim.

Reference analog: HRManagedPage.tsx — its Requests tab already renders Client Requests + Closed Requests stacked; the move makes Escalations the third stacked section using the same pattern. Risk

Level: 2 (Concern — isolated UI relocation, no data-layer change)

Risks Identified

  • Stale ?subtab=escalations deep links / bookmarks. After the tab is removed, a URL with subtab=escalations matches no tab-content conditional and would render an empty body. Mitigation: in each page, normalize an unknown/removed subtab value to the module's default tab (requests for HR/Payroll, pipeline for TA) when reading the param.
  • JSX bracket mismatch when removing the {activeTab === "escalations" && (...)} wrapper. Hand-editing nested JSX can drop a closing tag. Mitigation: rely on the TypeScript build gate (npx tsc --noEmit) plus a visual check after each file; edit one file at a time.
  • Spacing/empty-state placement. In HR/Payroll the Closed Requests table sits inside the "has data" branch of an empty-state ternary. Mitigation: place the moved Escalations section inside the same branch as the Closed Requests table so layout stays consistent, matching the existing mt-8 pattern.

Blast radius: Three page components only. No shared component, API, or type is modified. Consumers of getHREscalations/getPayrollEscalations/getTAEscalations are unchanged. EDC untouched.

Pushback

  • Stacking a third table in one tab makes the page longer and pushes escalations far below the fold. This is acceptable given the explicit request, and the kept section title keeps it findable. No better alternative within the stated goal.

Revision Log

Rev 1 - 2026-06-24

Change: Executed T1–T3. Managed HR was implemented exactly as specified — the Escalations CollapsibleSection was relocated inside the Requests tab's data branch, immediately after the Closed Requests table, wrapped in <div className="mt-8">.

Reason — Execution deviation: For Managed Payroll and Talent Acquisition the Escalations block was not physically adjacent to the target tab (the Reports tab sits between Requests and Escalations in Payroll; the Closed Positions tab sits between Pipeline and Escalations in TA). Rather than retype ~90 lines of table JSX (which carries exact-match risk, including an escaped "—" em-dash literal in Payroll), the existing Escalations block was re-gated from activeTab === "escalations" to the target tab ("requests" / "pipeline") and wrapped in <div className="mt-8">. Because the intervening tab block renders null when the target tab is active, the Escalations table appears at the bottom of the target tab exactly as intended.

Impact: Behavior matches the spec (Escalations at the bottom of the main Requests-style tab, tab button removed, table unchanged). The only difference from HR is structural: in Payroll/TA the Escalations table is a sibling block gated on the target tab rather than nested inside the data-branch fragment. One consequence: in Payroll/TA the Escalations table also renders during the target tab's loading/empty state; in HR it sits inside the populated data branch. Considered acceptable.

Stale-link guard: All three pages now normalize a removed/unknown ?subtab=escalations value to the module default (requests for HR/Payroll, pipeline for TA).
