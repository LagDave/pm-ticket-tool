Why

PMs need a home screen to manage their work. They come back to see past and in-progress interviews and act on them: resume an unfinished one, finish it, re-run it, or open the resulting ticket. Without this, a session that a PM leaves mid-interview is effectively lost — there is no way back into it.

This spec is the dashboard: the landing page that turns the persisted sessions from foundation (spec 1) and the resumable engine (spec 2) into something a PM can actually browse and act on.

What

A dashboard that lists the owner's interview sessions — each row showing status, a snippet of the original request, and when it was last updated — with four actions: open/resume (load the persisted state and route to the correct wizard step), finish (resume an awaiting-input session through to completion), re-run (clone the original_request into a fresh session — the locked default), and view ticket. Lists are paginated and owner-scoped. Resume reconstructs the wizard position by replaying interview_turns + decision_record (spec 2's mechanism), so a PM lands exactly where they left off.

Done when: a PM can list their sessions paginated, resume one mid-interview to exactly where they left off, re-run one as a fresh cloned session, and open a session's ticket. npx tsc --noEmit and npm run check:conventions --strict pass on both trees.

Patterns To Follow

  • Backend: list / session-state / clone endpoints mirror foundation's interview controller (§6.1, §7.1, §7.3); all DB in models, owner-scoped (§7.4, §11.7); standard pagination shape { page, limit, total, totalPages } (§11.6); envelope { success, data, error } (§8.1) via shared builders (§8.2); validation at the boundary (§11.2).
  • Frontend: triad Component → hook → api/ client (§14.1); the client is the only fetch path (§14.2 🔎); server state via React Query, UI state via Context (§15.1, §15.2), no extra state library (§15.4); errors via the one contract (§16.1).
  • Resume routes into the wizard (spec 2), reconstructing state from interview_turns + decision_record — do not re-derive interview logic here.

Reference analog: mirror foundation's backend triad (src/controllers/interview/ per §6.1) and frontend triad (api/ + React Query, §12.1), plus spec 2's resume/replay (rebuild wizard position from persisted interview_turns + decision_record).

Constraints

Must

  • Paginate all lists with the standard shape (§11.6).
  • Owner-scope every query in models/ by owner_user_id from server context (§11.7, §5.5).
  • Resume reconstructs the wizard position from the persisted interview_turns + decision_record (spec 2's replay).
  • Re-run is a clone: a fresh session seeded from original_request — the locked default.
  • Server state via React Query; UI state via Context (§15.1, §15.2); errors via the one contract (§16.1).

Must Not

  • Add a new state-management library (§15.4).
  • Mirror server data into useState (§15.1).
  • Reimplement the interview engine or ticket-generation logic — reuse specs 2 and 3.
  • Call fetch/axios outside api/index.ts (§14.2); put DB calls outside models/ (§7.4).

Out Of Scope

  • The interview engine and the ticket generator themselves (specs 2 and 3); any push-to-tracker integration.

Risk

Level: 2 (reads existing tables; no schema change; depends on specs 1–3).

Risks Identified

  • Resume correctness depends on the write-through schema from foundation / spec 2. If interview_turns + decision_record don't fully capture the wizard position, resume lands on the wrong step. Mitigation: rely solely on the persisted interview_turns + decision_record as the source of truth, and cover resume with tests that replay a partial session and assert the exact next step (§20.2).
  • Large session lists degrade the dashboard. Mitigation: pagination (§11.6) plus status filters on the list endpoint so a PM narrows by in-progress / completed.

Blast radius: reads interview_sessions, interview_turns, decision_record, and tickets; extends the existing interview controller/service/model and frontend triad. Depends on specs 1, 2, and 3; no new tables, no migration.

Pushback

  • "Finish" and "resume" can be one action. Finishing an awaiting-input session is just resuming it, then driving it to completion. Recommended: ship a single resume action that lands on the next unanswered batch — "finish" is simply resume on a near-complete session, not a separate code path. Two buttons in the UI are fine if they call the same resume route. Flag the simplification rather than building parallel logic.
