Why

UI-polish batch from PM review of the live app. The app has three different loading indicators (batch orb, triage gauge, inline line) which feel inconsistent; the PM likes the batch-orb style and wants it everywhere, with rotating "Dave the engineer" copy for personality. The PM added a new logo.webp to use over the PNG, needs a way to delete sessions (none exists today), and wants the dashboard's primary actions color-coded (New session = accent, Resume = white).

What

  • One shared ThinkingLoader (orange dual-ring orb + swapping serif title + optional subtitle + optional green-check "done" state) used for every loading state: batch generation, triage, session-list load, feature-scope load.
  • A pool of ~50 rotating messages, most about "Dave" the engineer, the rest about the codebase/process; shuffled per mount.
  • Header logo + favicon use /logo.webp (file moved from dist/ to public/ so it survives rebuilds).
  • Delete a session: owner-scoped DELETE /sessions/:id (DB cascade removes turns, decisions, tickets, comments, scout cache/jobs) + a per-row Delete with inline confirm on the dashboard.
  • Dashboard "New session" is the accent (orange) button; row "Resume"/"Finish" is a white button with black text.

Done when: all loaders are the single component with rotating copy; logo.webp serves in header + favicon; a session can be deleted and disappears from the list; the two buttons are recolored; tsc/lint/build/tests green; deployed and verified live.

Patterns To Follow

  • Backend mirrors gbp-automation/ (§6.1): route → controller → service → model; {success,data,error} envelope (§8.1); typed InterviewError + central status map (§8.3); every model query owner-scoped (§11.7, §5.5).
  • Frontend mirrors the api/ + React Query + feature-folder triad (§12.1): apiDelete in the one client, a thin domain fn, a mutation hook that invalidates ["sessions"], a presentational component (§14.1).
  • Loader is presentational + motion only (§14.1); rotating-message pool and timings are named constants (§4.2); no any (§17.2).

Reference analog: cloneSession end-to-end (model → service → controller → route → api → useCloneSession) is the closest existing CRUD-on-a-session path; the delete mirrors it.

Constraints

Must

  • Delete is owner-scoped at the model layer (§11.7); a foreign/missing id returns 404 via NOT_FOUND, indistinguishable (§5.5).
  • One loader component; the three old loaders removed, not left dead (§4.1, dead-export hygiene).
  • Move logo.webp into frontend/public/ (Vite source), not reference it from dist/.

Must Not

  • No new dependencies (lucide-react + framer-motion already present).
  • No DB migration - all session-child FKs already ON DELETE CASCADE; do not re-declare schema.
  • No accent creep: the orb is a genuine progress indicator; buttons stay restrained (only New session = accent, Resume = white).

Out Of Scope

  • Authentication (still the real gap; everyone is owner 1).
  • Soft-delete / archive / undo - delete is a hard cascade delete.
  • Bulk delete; delete from inside the wizard (dashboard rows only).

Risk

Level: 2 (Concern). One destructive endpoint; everything else is cosmetic.

Risks Identified

  • Irreversible data loss (delete cascades to tickets/comments/turns/decisions/scout rows). Mitigation: owner-scope at the model (§11.7) so no cross-owner delete; an inline two-step confirm in the row (not a one-click button); hard delete is the PM's explicit ask for their own data.
  • Cascade reaches more tables than obvious. Mitigation: verified every session_id FK is ON DELETE CASCADE (interview_turns, decision_record, tickets→ticket_comments, scout_cache, scout_jobs); the DB does it atomically, no app-side multi-write.
  • Loader regressions (StrictMode double-mount intervals, AnimatePresence exit deadlock seen before in BatchThinking). Mitigation: single setInterval cleared on unmount; render the loader directly without a wrapping mode="wait" presence (the prior fix).

Blast radius: loader change touches QuestionBatch, TriageBranch, Dashboard (3 consumers). Button/CSS change is dashboard-only. Delete adds one route; no existing endpoint changes.
