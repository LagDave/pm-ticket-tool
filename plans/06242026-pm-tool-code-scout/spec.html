Why

The interview only gives good options if it knows what already exists in the real codebase. This spec grounds it: given a session's request, a bounded, one-time scan finds the few relevant areas and summarizes what is already there, roughly how big the work is, and what it touches — so the next spec (grounded options, 6 of 8) can offer real choices instead of guesses.

The scan sits behind a pluggable provider so the source can change — GitHub now, Azure Repos later — without rewriting the scout. The code is read for orientation only: the scout never writes code-level or file-level implementation steps, because that is the engineering team's job, not the PM's. Findings are tagged "verify with engineering" so they are never mistaken for certainty.

What

Four pieces, mirroring foundation's backend triad (§6.1). (1) A CodeContextProvider interface — search code and read a file over a repo — so the source is swappable. (2) A GitHub provider implementing it, over the GitHub MCP server or the GitHub REST API, using a server-side token (§5.1, §17.3). (3) A bounded scout built on the Agent SDK, hard-capped on tool-calls, files, and tokens, that takes the request and returns structured findings: the relevant areas, "what exists / rough size / what it touches", and feasibility and effort hints. (4) A new scout_cache table that stores findings per session; the scout runs once per session and is read-through after that — later turns read the cache, they never re-scan.

Done when: for a session pointed at a repo, running the scout produces bounded, orientation-only findings tagged "verify"; a second request for the same session reads the cached row and does not re-scan; the scan is hard-capped; the repo token stays server-side; and npx tsc --noEmit plus npm run check:conventions --strict pass on the backend with no DB access outside models/ (§7.4) and no console.* (§9.1).

  • Mirror foundation's gbp-automation/ triad: thin controller (§7.3), business logic in feature-services/ (§7.1, §6.3), all DB in models/ (§7.4 🔎), layering Routes → Controllers → Services → Models (§7.1).

  • Repo token and scan caps come from config/ only, never literals (§4.2, §5.1); secrets stay server-side, never in the bundle (§17.3).
  • Pino, no console.* (§9.1 🔎); errors carry context — session id, provider, repo (§3.3); never leak internals to the client (§3.4).
  • Models + service tests cover the contract, error paths, and that the read-through cache returns the stored row without re-scanning (§20.1, §20.2).

Reference analog: mirror foundation's controller / service / model triad (§6.1, Constitution Part VI); ScoutCacheModel extends foundation's BaseModel; the scout_cache migration mirrors foundation's migration style (TODO-marked scaffold, Postgres-only).

Constraints

Must

  • Bound the scan hard — cap tool-calls, files, and tokens; it is a bounded scout, not a free-roaming agent.
  • Cache findings per session and serve read-through — run the scout once, read the cache on every later turn.
  • Keep the GitHub provider behind the CodeContextProvider interface so Azure plugs in later without touching the scout.
  • Keep the repo token server-side only, read from config (§5.1, §17.3); never ship it in the frontend bundle.
  • Keep findings orientation-only and tag them "verify with engineering" (§3.4 — no false certainty).
  • All scout_cache access in ScoutCacheModel (§7.4); owner-scoped through the session (§11.7).

Must Not

  • Implement the Azure Repos provider — that is a later spec.
  • Produce code-level or file-level implementation steps — orientation only, not the engineering plan.
  • Let the scout run per turn — it runs once per session and is read from cache thereafter.
  • Wire findings into the interview — that is the grounded-options spec (6 of 8).
  • Put DB calls outside models/ (§7.4); use console.* (§9.1); leave the scan unbounded.
  • Ship an mssql.sql migration — this app is Postgres-only (mirrors foundation).

Out Of Scope

  • The Azure Repos provider; wiring findings into the interview (spec 6); code-level implementation guidance; re-scanning on later turns.

Risk

Level: 3 (a bounded agent loop calling external repos, the longest operation in the app, behind a swappable provider).

Risks Identified

  • The agent loop is the longest operation and can exceed a Vercel serverless timeout. Mitigation: bound it hard (cap tool-calls/files/tokens) so it cannot run away, and run it as a separate extended-duration function or a queued background worker — the runtime is decided in the deploy spec (8 of 8); cache so it runs once per session and never blocks later turns.
  • GitHub MCP maturity, and Azure is deferred. Mitigation: the CodeContextProvider interface isolates the source choice; ship GitHub first (MCP or REST), add Azure later without touching the scout.
  • Wrong or stale findings could erode the PM's trust. Mitigation: tag every finding "verify with engineering" and present it as orientation, never as certainty (§3.4); keep findings coarse (areas, rough size, what it touches), not file-level claims.
  • Repo-token handling (a secret). Mitigation: read it from server-side env via config only (§5.1, §17.3), validate at startup (§5.6), never log it (§5.3), never expose it to the frontend.

Blast radius: writes the new scout_cache table and reads interview_sessions (foundation). Consumed by grounded-options (6 of 8), which reads cached findings to build interview options. No other consumers yet.

Pushback

  • Where does the scout point — which repo per session? The scout needs a repo reference, and the session does not carry one yet. Recommended: the session carries a repo reference (provider + repo identifier) set at scout time, stored so the cached findings record which repo they came from. The alternative — a single app-wide repo from config — is simpler but breaks the moment the tool covers more than one codebase. Recommend per-session; flag your call.
