Rich Ticket + Shareable Link

Status
Pending Execution
Plan Folder

Size
Large (~16 files + migration)

Level 3 (public endpoint)

Why

The PM answers many interview questions, but the finished ticket only renders four fields (user story, acceptance criteria, effort, context). Every recorded decision is fed to the generator, then compressed into one sentence plus a couple of criteria. The detail the PM worked to capture disappears. This is a schema-width problem, not a prompt problem: the generator can only emit four narrow fields. Second, a finished ticket cannot be handed to anyone outside the app, so the PM has no way to share it.

What

Two deliverables, shipped together:

  1. An enriched, full ticket. The generator and the renderer gain seven new sections so the PM's answers are visible in the artifact, not flattened away.
  2. A short, copyable version with a back-link. The copy action produces a compact Markdown block (core fields + a View full ticket link). The link carries an unguessable share token and opens a public, read-only view of the full ticket in the app, for anyone the PM pastes it to.

Done when: a generated ticket shows all sections in-app; the copy button yields a short block ending in a working link; opening that link in a logged-out/other context renders the full ticket read-only; the public endpoint never exposes comments or owner data; the backend type-checks, lints, and tests pass.

Patterns To Follow

  • Backend reference analogs (§6.1): the existing ticket domain (TicketService/TicketController/routes/tickets.ts) and gbp-automation/. The public-by-token read mirrors InterviewSessionModel.findByIdSystem - a deliberate non-owner-scoped read on a trusted/capability path (§21 note), distinct from the owner-scoped reads.
  • Frontend reference triad (§12.1): api/tickets.ts + useTicketQueries.ts + components/ticket/. The shared view mirrors TicketView; extract the read-only render into a shared TicketReadView so both views use one renderer (§4.3).
  • Migration analog: src/database/migrations/20260624000002_ticket_draft_schema.ts - TS Knex, alterTable + raw CHECK constraints, Postgres-only by design.
  • No router (§15.4): the deep link is read once from window.location.search on mount in App.tsx and seeds the view, mirroring the existing useState view switch. No router library.

Reference file: src/controllers/ticket/feature-services/TicketService.ts - closest analog for the new SharedTicketService (owner-verify shape inverted to token-lookup).

Constraints

Must

  • All DB access through models/ (§7.4); layering Routes -> Controllers -> Services -> Models (§7.1).
  • { success, data, error } envelope (§8.1) via the existing response builders; typed TicketError + central status mapping (§8.3).
  • Validate the token param at the route boundary (§11.2); rate-limit the public router (§11.3).
  • Share token from crypto.randomBytes(32) base64url - 256-bit, unguessable (§5.1, §5.2). Byte length is a named constant (§4.2).
  • Public DTO exposes ticket content only - never comments, owner ids, or session internals (§5.4, §5.5).
  • Boundary schema requires only story + >=1 criterion; new fields are best-effort (empty arrays/strings allowed) so a thin model answer never fails generation (§11.2).
  • No any - type the details jsonb and the public DTO (§4.5, §17.2). Frontend fetch only through api/index.ts (§14.2); fetching in a hook (§14.3); server state in React Query (§15.1).
  • Tests mirror each new unit and cover the contract, the not-found path, and that the public read leaks nothing (§20.1, §20.2).

Must Not

  • No router library on the frontend (§15.4).
  • No env-specific URL baked into rendered_markdown or the DB - the short link is built client-side.
  • No second state system; no raw fetch/axios outside the client (§14.2).
  • Do not make the rich generated arrays editable in v1; do not touch unrelated domains.
  • Do not widen the public endpoint to list/enumerate tickets - single-token lookup only.

New Dependency

  • express-rate-limit (pinned) - justified by §11.3 for the one unauthenticated public route; applied only to the shared router. Flagged per §4.4; cut at review if the 256-bit token entropy is deemed sufficient and rate-limiting is deferred to the auth/hardening spec.

Out Of Scope

  • Real authentication (the known next priority; ownerContext stays the dev placeholder).
  • Editing the generated rich arrays (Key Decisions, Open Questions, etc.).
  • Token rotation/revocation, share analytics, expiry.
  • Sharing comments or any collaborative/public write.

Risk

Level: 3 - a new unauthenticated public endpoint.

Risks Identified

  • Public endpoint exposure (L3). GET /api/shared/tickets/:token is unauthenticated. Mitigation: 256-bit crypto-random token (brute-force infeasible) + minimal content-only DTO + boundary token-format validation + rate-limit on the router. The row carries no owner id; comments are never fetched.
  • New dependency (L2). express-rate-limit. Mitigation: pin it, scope it to the shared router; documented justification (§11.3).
  • Data-model choice (L2). Six rich fields in one details jsonb vs six columns. Mitigation: jsonb (we never filter on them); share_token/priority stay first-class. Tradeoff: slight divergence from the per-column style of acceptance_criteria.
  • Token staleness on regenerate (L2). A regenerated ticket is a new row with a new token. Mitigation: accepted - edits/finalize mutate the same row, so a shared link stays current; regeneration is not exposed in the UI.
  • Larger model output (L2). More fields = more tokens, higher malformed-field odds. Mitigation: modest MAX_TOKENS bump; best-effort fields; boundary still only hard-requires story + one criterion.
  • AI-guessed priority (L2). Unreliable like hours. Mitigation: coarse tier + editable + "confirm with team" note, mirroring effort.
  • Two renderers (L2). Full (server) vs short (client). Mitigation: the short is a deliberately minimal artifact (4 fields + link), not a re-implementation of the 11-section full renderer (§4.3).

Blast radius: GeneratedTicket/ITicket/Ticket(FE) widen (additive). Consumers re-checked: ticketAgent, TicketGenerationService, TicketMarkdownService (all render paths), TicketService (edit/finalize), TicketView, TicketEditFields, CopyTicketButton, and their tests. tickets migration backfills tokens for existing rows. No existing route path collides (verified: API under /api, catch-all skips it).

Pushback

  • Shipping a public unauthenticated endpoint before real auth is a deliberate tradeoff for shareability. It is defensible because the token is the capability and the DTO is content-only - but it widens the public surface of an app that already has no login. Recommended: proceed with the four mitigations; revisit token rotation/expiry when the auth spec lands.
  • The generic plan convention asks for mssql.sql too; this project is Postgres-only by design (see the analog migration). Recommended: ship knexmigration.ts + pgsql.sql only; MSSQL N/A.
