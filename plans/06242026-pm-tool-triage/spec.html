Why

Most tickets do not need the full interview. A bug report or a one-line copy change is wasted on a multi-turn questioning loop, and that friction is exactly what makes PMs skip the tool. This adds a quick triage step right after request entry that splits requests into two speeds: simple asks go straight to a draft ticket (with at most a question or two), and only genuinely scoped work is routed into the full interview loop — so the heavy machinery stays out of the way for the common case.

Triage sits between the foundation's request-entry screen (spec 1) and the two paths it feeds: the ticket-draft generator (spec 3) and the interview engine (spec 2). It is a router, not a new engine — it reuses both paths rather than reimplementing either.

What

A triage step that runs after request entry. A cheap, low-effort Agent SDK classification labels the request simple or scoped. A simple request goes to the ticket-draft path (spec 3) with 0–1 clarifying questions; a scoped request enters the full interview loop (spec 2). The classification is always overridable — the PM can force the full interview from a simple result, or skip straight to drafting from a scoped result. The result is persisted on the session via the InterviewSessionModel.

Done when: a simple request skips the interview and still yields a valid standard-format ticket, a scoped request enters the full loop, the PM can override either way, the triage_result is persisted on the session, and npm run check:conventions --strict plus npx tsc --noEmit pass on both trees.

Constraints

Must

  • Conform to the Constitution; cite §N.M on any deviation.
  • Triage is cheap and fast — a low-reasoning-effort Agent SDK call, not the full question generator.
  • The simple path still produces a valid standard-format ticket (story + acceptance criteria + effort), via spec 3.
  • The classification is always overridable; default to scoped whenever the model is unsure (§5.4 — the client never decides scope authoritatively).
  • Persist the result through InterviewSessionModel, owner-scoped (§7.4, §11.7); secrets server-side only (§5.1).

Must Not

  • Replace the interview for scoped work — reuse the interview engine (spec 2).
  • Make triage a hard gate; always allow the PM to override the label either way.
  • Reimplement question generation or ticket drafting — reuse specs 2 and 3.
  • Put the triage write outside models/ (§7.4); add a second state library on the frontend (§15.4).
  • Ship the Agent SDK key to the bundle — server-side only, never VITE_ (§17.3).

Out Of Scope

  • The interview engine and the ticket generator themselves (reuse specs 2 and 3); the code scout (spec 5); grounded options (spec 6); the dashboard (spec 4).

Risk

Level: 2 (small additive change; one new service, one model method, one schema column, two existing paths reused).

Risks Identified

  • Misclassification — a scoped request labelled simple, or the reverse. Mitigation: the PM override is always available, and the classifier defaults to scoped whenever it is unsure, so the failure mode is "too much interview," never "a thin ticket from scoped work." Override and default-to-scoped are both required, not optional.
  • An over-eager simple path produces thin tickets. Mitigation: the simple path may still ask 0–1 clarifying questions before drafting, and the draft still goes through spec 3's standard ticket format, so a simple ticket is short but never malformed.
  • Triage adds an Agent SDK call to every request, raising latency and cost. Mitigation: keep it a single low-reasoning-effort call with a tight prompt and a small expected output; do not stream or loop. Secrets stay server-side (§5.1, §17.3).

Blast radius: adds a triage_result column (and optional triaged_at) to interview_sessions; routes into spec 2 (interview engine) and spec 3 (ticket draft); depends on specs 2 and 3 being in place. No existing consumer of interview_sessions reads these new columns yet, so the column add is non-breaking.

Pushback

  • Heuristic vs LLM triage. Recommended: a low-effort LLM classification with a deterministic override, because request phrasing is too varied for a reliable keyword heuristic and the override plus default-to-scoped cap the downside. Alternative if per-request cost is the dominant concern: a pure heuristic (keyword/length rules) with the same override — cheaper and deterministic, but it will misroute more, pushing more genuinely-simple asks into the full interview. Flag your call before execution.
