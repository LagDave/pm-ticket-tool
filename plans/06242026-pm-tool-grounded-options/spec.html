Why

This makes the interview's options real. In the interview engine spec (2 of 8) the options on each generated question are stubbed — generic brainstorm, not anchored to anything the codebase can actually do. Here we feed the cached scout findings (5 of 8) into question generation so each option is grounded in what is buildable, each carries an effort tier, and one can be flagged as recommended. That recommended pick is the "here's the easier way" payoff the tool exists to deliver.

The same findings also let the materiality gate skip questions the code already answers. If the scout already found, say, that auth is handled by an existing provider, the PM should not be asked to choose one. Asking the codebase first instead of the human is the discipline that keeps the interview short and honest.

What

Extend the interview engine spec's batch-generation service to read the session's scout_cache findings and, for each question it emits, produce options that derive from those findings — each tagged with an effort tier and an optional recommended flag — and to skip questions whose answer the findings already determine. The option shape gains effort and recommended fields inside the existing interview_turns.questions jsonb; there is no schema change and the scout is not re-run (read the cache only). When a session has no findings (no repo, or the scout was skipped), generation falls back to ungrounded options cleanly rather than failing.

Done when: for a session with cached findings, generated batches carry grounded options with effort tags and a recommended pick, questions the findings already answer are skipped, no-findings sessions still generate ungrounded options, no migration is added, and npm run check:conventions --strict plus npx tsc --noEmit pass on both trees with no any (§17.2).

  • src/controllers/interview/feature-services/MaterialityGateService.ts — the gate from spec 2; extended here to skip questions the findings already answer (codebase-first). Business logic only, never touches the DB (§7.1).

Patterns To Follow

  • Backend layering Routes → Controllers → Services → Models (§7.1); findings are read through ScoutCacheModel only, never an inline query (§7.4 🔎).
  • Reuse the persisted interview_turns.questions jsonb established in foundation — the new option fields live inside it; no migration.
  • Frontend triad Component → hook → api/ client (§14.1); the question-batch hooks are unchanged (§14.3, §15.1) — this spec only enriches the rendered option shape.
  • Errors surface through the one contract end to end (§16.1); no any on either tree (§17.2 🔎); secrets stay server-side — the Agent SDK call runs in the service, not the bundle (§17.3).

Reference analog: backend mirrors foundation's src/controllers/interview/ generation service (built to gbp-automation/, §6.1, Constitution Part VI) consuming the code scout spec's ScoutCacheModel (§7.4); frontend mirrors foundation's frontend/src/api/ + React Query triad (§12.1). Note: the generation service and the cache come from specs 2 and 5 — this spec extends them, it does not create them.

Constraints

Must

  • Conform to the Constitution; cite §N.M on any deviation.
  • Options derive from the cached findings, not generic brainstorm — each option points back to what a finding supports.
  • Effort is a tier (a small fixed enum, e.g. low / medium / high), not free text or an hour estimate.
  • Exactly one option per question may carry the recommended flag; it is optional, not forced.
  • Skip a question when the findings already determine its answer (the materiality gate, codebase-first).
  • Read findings through ScoutCacheModel only (§7.4); validate the model's structured output at the boundary (§11.2); no any (§17.2).

Must Not

  • Change the persistence schema — reuse interview_turns.questions jsonb; no migration in this plan.
  • Re-run the scout or call any repo provider — read the cache only.
  • Present grounded claims as certain — options carry a "verify with engineering" tone, never a guarantee.
  • Reimplement the interview loop or batch persistence — those are the interview engine spec (2 of 8).
  • Add a second state library on the frontend (§15.4) or a raw fetch outside the api client (§14.2).

Out Of Scope

  • The code scout itself and any repo provider (5 of 8); the interview loop mechanics, turn persistence, and answer capture (2 of 8); triage routing and push-to-tracker (7 of 8).

Risk

Level: 2 (enriches an existing generation path; reuses the jsonb; no schema or consumer break).

Risks Identified

  • Wrong grounding misleads the PM into a false sense of certainty. Mitigation: keep options to what the findings actually support, and frame every grounded claim with a "verify with engineering" tone in both the generated copy and the rendered option — the tool advises, it does not certify.
  • Option quality drifts over time (grounded output regresses toward generic brainstorm). Mitigation: a structured option schema (effort enum + recommended boolean, validated at the boundary) plus tests that compare grounded output against ungrounded output for the same prompt and assert the grounded set references findings (§20.2).
  • Over-eager skipping hides a question the PM still needs to answer. Mitigation: the gate skips only when a finding fully determines the answer; record the skip reason so it is auditable, and cover the threshold with a test (§20.2).

Blast radius: changes how the interview engine spec (2 of 8) generates options and how it reads the code scout spec's scout_cache (5 of 8) — it depends on specs 2 and 5 and must land after both. The reused interview_turns.questions jsonb is unchanged in shape at the DB level, so foundation's migrations and models are untouched.

Pushback

  • No-findings fallback must be explicit, not incidental. If a session has no repo connected or the scout was skipped, ScoutCacheModel returns nothing and generation must produce ungrounded options on a clean, named path rather than erroring or emitting empty option lists. Recommended: branch on "findings present" once in QuestionBatchService and route to a grounded or ungrounded generation path, both returning the same option shape (effort/recommended simply absent when ungrounded). Flag if you would rather block generation until a scout has run — that couples specs 2 and 5 harder than the rest of the plan assumes.

Rev 1 - 2026-06-24

Change: Executed the spec. Four deviations from the spec text, all recorded below; the contract (grounded options with effort + a recommended pick, codebase-first skip with audit, clean no-findings fallback, no migration) is met as written.

Reason — Execution deviation (file paths, §6.1): The spec named QuestionBatchService.ts, feature-utils/questionSchema.ts, and QuestionBatch.tsx. None exist — they were placeholder names from before the interview engine (spec 2) landed. The real generation path is src/agents/interviewAgent.ts (the only place the SDK is constructed, §6.2) orchestrated by InterviewEngineService.ts, with the structured-output schema in src/validation/interviewQuestions.ts. Per §6.1 (mirror the reference implementation) the work extended those real files. The option grounding fields were already stubbed there for exactly this spec ("filled by a later grounding step").

Reason — Execution deviation (skip location, §2.1): T2 asked MaterialityGateService to filter individual questions. That service is round-level pure logic ("generate another batch or terminate"); per-question filtering is a different responsibility (§2.1). There is also no structured link between a question's decisionKey and a finding's area, so code-side matching would be a fuzzy-string heuristic — exactly the over-eager-skip risk the spec flags. Instead the model performs the skip in the grounded prompt (it has both the findings and the questions), gated on a finding's feasibility === "clear", and returns a structured skipped[] array. InterviewEngineService wires the findings in and logs the skips; the array persists inside the interview_turns.questions jsonb (auditable, replays on resume). MaterialityGateService is untouched.

Reason — Execution deviation (effort field, §4.3): The option shape gained a new effort enum (XS|S|M|L|XL) and a recommended boolean; the pre-existing groundingRef stub was kept and filled. The stub effortTag: string|null was dropped in favour of the typed effort tier. The enum reuses EFFORT_HINTS from validation/codeScout.ts (its declared source of truth) rather than re-declaring the literal (§4.3), so a grounded option speaks the same tier vocabulary as the finding's roughSize. The "at most one recommended per question" rule is enforced by a refine on the boundary schema, mirroring the existing question-cap refine (§11.2).

Reason — Execution deviation (test location, §20.1): T4 listed the skip test under MaterialityGateService.test.ts. Because the skip moved out of that service, the grounded / ungrounded / skip / over-recommended tests live in InterviewEngineService.test.ts (mirroring the unit they cover, §20.1). MaterialityGateService.test.ts is unchanged.

Rev 2 - 2026-06-24

Change — User QA: Replaced the option-level effort tier with an ordered build-speed scale and tightened the recommended rule to exactly one per question. The option contract is now { id?, label, speed, recommended, groundingRef? }.

Reason — User QA (speed scale, §4.5/§11.2): The Rev 1 effort enum (XS|S|M|L|XL, reused from the scout's EFFORT_HINTS) was replaced by a dedicated ordered 5-step scale speed: "slowest" | "slow" | "moderate" | "fast" | "fastest" — slowest = most work to build, fastest = least. It is a standalone per-option build-speed indicator, distinct from the scout's roughSize and the ticket-level EffortTier, so it gets its own SPEED_TIERS source of truth in src/validation/interviewQuestions.ts (the EFFORT_HINTS import there was dropped; the scout's own use is untouched). The closed-set enum stays structured-output-safe, so the generation and boundary schemas remain identical (§11.2). The shared QuestionOption type was updated in both src/types/interview.ts and frontend/src/types/interview.ts (no any, §4.5/§17.2).

Reason — User QA (exactly-one recommended): recommended is now a plain boolean (no longer nullable) and every option carries one; the generation prompt instructs the model to mark exactly one option per question recommended: true — the single best option — on both the grounded and the ungrounded (no-findings) paths. The boundary ≤1-recommended refine is preserved as-is (zero is still schema-valid; the "exactly one" expectation is enforced in the prompt), per the existing question-cap refine pattern.

Reason — User QA (prompt + fallback): Both prompts in src/agents/interviewAgent.ts were updated: the old EFFORT_HINTS-style guidance became a SPEED_DESCRIPTIONS map rendered as an ordered slowest < … < fastest line, and a shared COMMON_OPTION_RULES block (speed tier + exactly-one-recommended) now applies to both paths via buildSystemPrompt. The ungrounded fallback stays intact — no groundingRef, skipped: null — but its options now also get a speed tier and one recommended pick.

Reason — User QA (frontend, minimal): QuestionField.tsx and QuestionBatch.tsx were adjusted minimally to reference speed instead of effort and to compile/render a basic Speed: {speed} label; the grounded-signal heuristic now keys on groundingRef (since speed/recommended are present on every option). No visual redesign — a separate agent overhauls this UI next.
