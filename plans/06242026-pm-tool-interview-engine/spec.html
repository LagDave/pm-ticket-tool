Why

This is the heart of the product. A PM types a vague request; this engine turns it into complete, consistent requirements through a batched, adaptive interview that always terminates. It generates a few dependency-ordered questions at a time, records the PM's answers as structured decisions, and stops once it has enough to write a good ticket — never an endless wall of questions.

It must be resumable. Every step writes through to foundation's interview_turns and decision_record tables, so a PM can leave mid-interview and the dashboard (spec 4) can resume or finish the session later from saved state. Grounding the questions in the real codebase comes later (spec 6); here the interview is grounded on the request text only.

What

A server-side TypeScript Agent SDK loop that: (1) generates a batch of up to four dependency-ordered questions — each with answer options (option grounding and effort tags are stubbed here; real grounding is spec 6) — via a structured-output schema; (2) persists the batch to interview_turns; (3) accepts the PM's answers and writes structured rows to decision_record with source = answer | default; (4) runs a materiality gate that generates another batch only if an open decision would change the ticket; (5) loops under a hard max-rounds cap; (6) terminates at shared understanding. Each question supports an "other / free-text" answer, and the PM can hit a global "stop and generate now." Resume means replaying persisted turns plus the decision record — no batch is ever regenerated.

Done when: a vague request runs a multi-round interview to termination, every batch and answer and decision is persisted, the session resumes correctly from any saved mid-point, and npx tsc --noEmit plus npm run check:conventions --strict pass on both trees. No ticket generation (spec 3), no dashboard browsing (spec 4), no repo grounding (spec 6).

Patterns To Follow

  • Agent SDK logic in src/agents/ (§6.2), wrapped by a feature-service under controllers/interview/feature-services/ (§6.3); layering Routes → Controllers → Services → Models (§7.1); thin controller (§7.3); all DB through models (§7.4 🔎).
  • Structured-output schema with validation at the boundary on malformed model output (§11.2); response envelope (§8.1) + typed InterviewError with one status mapper (§8.3).
  • Multi-table writes (a turn plus its decision rows) run in one transaction (§10.5); Pino, no console.* (§9.1 🔎); secrets server-side, only VITE_ in the bundle (§5.1, §17.3).
  • Frontend triad Component → useInterview hook → api/ client (§14.1); the client is the only fetch path (§14.2 🔎); server state is React Query (§15.1).

Reference analog: mirror foundation's interview controller/service/model triad — built to src/controllers/gbp-automation/ (§6.1, Constitution Part VI) on the backend and the frontend/src/api/ + React Query triad (§12.1) on the frontend. This spec extends that established analog; it does not invent a new one.

Constraints

Must

  • Write-through persistence on every step — each batch and each answer set is durable before the next step, in a transaction (§10.5).
  • A materiality gate plus a hard max-rounds cap — together they kill the "goes on and on" failure mode.
  • Persist structured decision_record rows (key/value/source), not chat text.
  • Read ANTHROPIC_API_KEY via config, server-side only (§5.1, §17.3).
  • Low reasoning effort on generation calls to control cost; resume must replay saved batches, never regenerate.
  • Validate the structured question output and the submitted answers at the boundary (§11.2).

Must Not

  • Ground questions on the repository — that is the code-scout / grounded-options spec (6).
  • Generate the final ticket — that is the ticket-draft spec (3).
  • Build dashboard browsing / resume UI — that is the dashboard spec (4).
  • Add database tables — reuse foundation's interview_turns and decision_record.
  • Put DB calls outside models/ (§7.4); use console.* (§9.1); ship the API key in the Vite bundle (§17.3).

Out Of Scope

  • Code scout, option grounding and effort tags (real values), triage routing, and ticket rendering.

Risk

Level: 3 (the engine is the product's core; non-termination, cost, and serverless timeouts are real traps).

Risks Identified

  • Agent SDK long-running loop vs Vercel serverless timeouts. Mitigation: each batch generation is one bounded, short-lived API call, not a long agent loop; state persists between turns so no single HTTP request runs long; the heavy code scout is a separate spec and runtime. The final runtime/timeout tuning call is made in the deploy spec (8).
  • Non-termination — the interview "goes on and on." Mitigation: the materiality gate generates another batch only when an open decision would change the ticket, backed by a hard max-rounds cap and the global "stop and generate" escape.
  • LLM cost and latency. Mitigation: low reasoning effort on generation calls; persist every batch so resume replays saved state and never regenerates.
  • Malformed question output from the model. Mitigation: a structured-output schema plus validation at the boundary (§11.2); reject or repair off-schema output before it is persisted.

Blast radius: consumes foundation's interview_sessions / interview_turns / decision_record; later consumed by the dashboard (4), grounded-options (6), and triage (7). No tables are altered.

Pushback

  • If a per-session round count or a richer status is genuinely needed, prefer reusing foundation's existing columns over a schema change. Only add a small alter migration if it is unavoidable — and then mirror foundation's migrations style (Postgres-only: pgsql.sql + knexmigration.js, no mssql.sql). Flag the call before adding a column.
