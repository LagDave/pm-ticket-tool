Why

The interview is ungrounded: it invents app specifics (a "Puzzle page" that does not exist) and asks questions whose answer is already settled (web vs. mobile, when the app is web-only). Both come from the engine having no model of the app. The code scout was meant to supply that, but it is being abandoned (provider/token infra, per-interview latency, ~25 files). This replaces the scout with a human-curated, self-refreshing project context — a set of typed "bits" about the app — that grounds the interview (and the ticket) and stays current as work completes.

What

A Project owns a set of bits (typed key→summary rows). Bits are rendered into the interview's cached system prefix so the engine grounds options in real app facts and suppresses questions a settled fact already answers. Bits also feed the ticket generator. They are maintained three ways: (1) manual CRUD; (2) bulk import of a JSON file produced by a separate Claude Code session via a copy-paste generate-bits prompt; (3) merge-on-complete, which turns a finished ticket into candidate bits. A reconciliation agent dedups/merges/flags conflicts on every add (import or merge), with a human resolve step. The entire code-scout subsystem is deleted.

Done when: a session attached to a project produces options grounded in that project's bits, skips questions settled by constraint/tech_stack/inventory bits (visibly, with an "answer anyway" override), the import + reconcile + resolve loop works end-to-end, completing a ticket proposes bits, and the scout code/tables are gone with a green tsc/test run.

Context

The grounding seam (the surgical core)

  • src/controllers/interview/feature-services/InterviewEngineService.ts — findCachedFindings(sessionId) (reads scout_cache) is the only live scout read; it feeds generateAndPersistBatch. Replace with a project-bits loader keyed off session.project_id. The grounded/ungrounded branch carries over.

  • src/agents/ticketAgent.ts + src/controllers/ticket/feature-services/TicketGenerationService.ts — ticket generation gets {originalRequest, decisions} only; add project bits.

Key decisions carried from planning

  • Bit keys are cosmetic labels; dedup is semantic. No unique (project_id, bit_key) constraint — the reconciliation agent matches by meaning, not key (resolves key-drift).
  • Merge-default, never auto-delete. The agent proposes; insert auto-applies, everything else is human-resolved; force overrides; status/source keep an audit trail.
  • Import is additive. Never clears existing bits unless an explicit force/replace flag is set.
  • Suppression is visible. Skipped questions render with their reason + an "answer anyway" override (R3).
  • Ticket grounding = yes (no-preference → locked; veto in -c).

Constraints

Must

  • All DB access in models/ (§7.4); Routes → Controllers → Services → Models (§7.1); thin routes (§7.2).
  • {success,data,error} envelope + typed ProjectError + one status mapper (§8.1–§8.3).
  • Owner-scope every project/bit query by OwnerContext as a required arg (§11.7/§5.5).
  • Validate every input (incl. the uploaded import file) at the route boundary via validation/ zod schemas (§11.2/§5.2).
  • Reconciliation agent + ticket/interview grounding = bounded structured-output calls with model+fallback (§6.2); re-validate at the boundary (§11.2).
  • Named constants in config/ — bit caps, import caps, reconciliation model/effort/tokens (§4.2).
  • Multi-row writes (apply plan, merge-on-complete) in a transaction (§10.5); index project_id / owner columns (§10.4).
  • Frontend: one HTTP client, React Query for server state, feature folders, one error contract (§12.1/§14.2/§15.1/§16.1); no any/console.* (§17.1/§17.2).
  • Bits render into a byte-stable cache_control system block; per-call data stays in the user message (R4).
  • Tests mirror each model/service, covering the contract, error paths, and tenant scope (§20.1/§20.2).

Must Not

  • No new runtime dependency (no vector DB / embeddings — bits ship whole in the prompt).
  • No auto-delete or auto-overwrite of bits by the agent; no clearing on import without force.
  • No per-call/volatile data in the cached system prefix (would break caching).
  • No inline db() outside models; no hand-rolled responses; no knex.raw outside the migration.
  • Do not delete scout files until the grounding swap (T5) has replaced the scout read.
  • Do not introduce real auth here — inherit the existing ownerContext seam unchanged.

Out Of Scope

  • Real authentication (still the dev ownerContext placeholder).
  • Azure repo provider / any code-scanning (the scout is removed, not replaced).
  • Embeddings/RAG over bits (explicitly rejected — bits fit the context window whole).
  • Auto-applying agent-proposed merges/conflicts without human confirmation.
  • Re-feeding existing bits into the generate-bits prompt for delta-only export (import dedup handles overlap).

Risk

Level: 3 (structural — scout teardown blast radius + an agent that can silently corrupt grounding).

Risks Identified

  • R1 · Scout teardown blast radius (L3). ~25 files, app.ts/index.ts/config unwiring, two applied tables dropped. Mitigation: sequence teardown last (Wave 3) after T5 replaces the scout read; rely on the tsc gate for dangling imports; drop tables in a dedicated migration (data abandoned by design; down() recreates empty schema).
  • R2 · Reconciliation can silently corrupt context (L3). A bad auto-merge degrades every future interview. Mitigation: agent only proposes; insert auto-applies, update/conflict/similar require human resolve; force explicit; status='superseded' + source retain an audit so a bad merge is reversible; the resolve screen must make conflicts cheap to scan.
  • R3 · Suppress + stale bit = silently dropped question (L2, accepted). Mitigation: only settled kinds suppress; skipped questions render visibly with reason + "answer anyway" override; skip is conditional on the request not being about changing that fact.
  • R4 · Caching only pays if the prefix is byte-stable (L2). Mitigation: bits + static rules in the cached system block, per-call data in the user message; verify via usage.cache_read_input_tokens; sub-4096-token projects simply don't cache (no harm).
  • R5 · Import file is untrusted model-generated JSON (L2). Mitigation: zod validation at the boundary, size/count caps, per-row error reporting, reject malformed; additive-merge default.

Blast radius: InterviewEngineService + interviewAgent (grounding swap), TicketGenerationService + ticketAgent (ticket grounding), app.ts + index.ts + config/index.ts (scout unwire), interview_sessions table (new project_id), the frontend interview wizard (picker + skipped surface), and the deletion of agents/codeScout/*, controllers/codeScout/*, services/codeContext/*, services/scoutJobs/*, workers/scoutWorker.ts, models/Scout*, routes/codeScout.ts, routes/internalScout.ts, types/codeScout.ts, validation/codeScout.ts, scripts/scout-work.ts (+ tests), and dropping scout_cache/scout_jobs.

Pushback

  • This was scoped as one Large plan by choice; the grounding pipeline alone fixes the screenshots, while the reconciliation agent is the hard, unverifiable part. Mitigation: execute in 3 waves so value lands in Wave 1 and the risky/fuzzy work is staged after a green, grounded baseline.
  • Suppression was chosen over pre-answer despite the silent-omission risk. If R3's visible-skip + override proves noisy in use, fall back to pre-answer (options still shown, known answer pre-recommended) — recorded as the v2 lever.
