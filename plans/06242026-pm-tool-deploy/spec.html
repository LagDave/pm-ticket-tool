Why

Specs 1 through 7 build the whole tool against a local Docker Postgres on host port 8765; nothing is live yet. This spec ships it. It is the last plan in the eight-spec sequence and runs only after the other seven are executed, because it depends on the real schema, the real interview engine, and the real code scout existing first.

One spec provisions production, wires continuous deployment, runs the existing migrations against Neon, and proves the live app works end to end — so the tool goes from "works on my machine" to "live on a URL, deploying on push" in a single, ordered step.

What

Provision a Neon Postgres database for production and capture its pooled, SSL DATABASE_URL. Configure a Vercel project that serves the React 19 + Vite SPA as a static build and the Express API, connected to the GitHub repo so a push to main auto-deploys. Set the production env vars in Vercel — DATABASE_URL (Neon pooled + SSL), ANTHROPIC_API_KEY, and the GitHub repo token the code scout (spec 5) uses — all server-side only, none in the Vite bundle. Run every existing migration (from specs 1, 3, 5, 7) against Neon before any traffic. Decide and document how the TypeScript Agent SDK runtime fits Vercel's serverless model.

Done when: the app is live on a Vercel URL backed by Neon, a push to main auto-deploys, all migrations are applied to Neon, the secrets are set in Vercel and absent from the shipped bundle, and a live end-to-end smoke test passes (health, create a session, run an interview, generate a ticket). No new product features and no new tables — this spec runs the schema that already exists.

Context

Provisioning happens at execution time against the user's own accounts. Neon, Vercel, and GitHub resources are created during execution, driven through the browser or CLI with the user present. This plan document touches no external account; it is the script the execution follows.

Relevant Files

  • vercel.json — Vercel build + routing config: the Vite static build for the SPA and the catch-all route to the Express API handler (created in T3).
  • src/index.ts — Express entry from foundation; startup config validation must fail fast on a missing or malformed DATABASE_URL / ANTHROPIC_API_KEY in production (§5.6).
  • src/config/index.ts — env access from foundation; production reads Vercel-injected env, never a committed file (§5.1).
  • src/database/connection.ts — the one central Knex pool from foundation; production must use the Neon pooled connection string with SSL enabled (§10.6).
  • knexfile.ts — the migration runner config; its production target reads the same DATABASE_URL so the existing migrations apply unchanged (§10.3).
  • src/database/migrations/* — the migrations authored by specs 1, 3, 5 (scout_cache), and 7; this spec runs them against Neon, it does not add to them.
  • frontend/.env.production / Vercel project env — only VITE_-prefixed public config (e.g. the API base URL); no secret ever ships here (§17.3).
  • plans/06242026-pm-tool-deploy/test.html + test-results.json — the Layer 2 acceptance artifact generated when this plan is executed (§20.5).

Patterns To Follow

  • Neon is serverless Postgres: every serverless caller uses the pooled connection string with SSL, through the one central Knex pool — never a per-request connection (§10.6).
  • Configuration is validated at startup and fails fast with a clear message; a missing or malformed production env var is caught at boot, not at request time (§5.6).
  • Secrets live in Vercel env only and are accessed via config; only VITE_ public config is shipped to the browser (§5.1, §17.3).
  • CORS is configured explicitly for the deployed Vercel origin (no wildcard in production) and Helmet stays enabled on the API (§11.4).
  • Schema reaches production only through the existing Knex migrations, run against Neon before first traffic (§10.3).

Reference analog: there is no deployment analog in the repo — this is the first production cut. It mirrors foundation's connection, config, and security choices: the central Knex pool (§10.6), startup config validation (§5.6), and the explicit CORS + Helmet setup (§11.4) that foundation established. The Agent SDK runtime is server-side TypeScript; its routes live behind the same Express app foundation built.

Constraints

Must

  • Keep secrets in Vercel env only, accessed via config (§5.1, §17.3).
  • Use the Neon pooled connection string with SSL from every serverless caller, through the central pool (§10.6).
  • Run all existing migrations against Neon before any production traffic (§10.3).
  • Auto-deploy on push to main via the Vercel + GitHub integration.
  • Validate production config at startup and fail fast on a missing or malformed value (§5.6).
  • Configure CORS explicitly for the deployed origin; keep Helmet enabled (§11.4).

Must Not

  • Commit any secret — no DATABASE_URL, ANTHROPIC_API_KEY, or repo token in the repo or the bundle (§5.1).
  • Use an unpooled Neon connection from serverless functions (§10.6).
  • Expose ANTHROPIC_API_KEY or the GitHub repo token to the SPA — nothing sensitive behind a VITE_ prefix (§17.3).
  • Add tables, alter the schema, or write new migrations — this spec runs the schema specs 1/3/5/7 already defined.
  • Run manual DDL against Neon instead of the migrations (§10.3).

Out Of Scope

  • Adding product features; the Azure repo provider; push-to-tracker.

Risk

Level: 4 (outward-facing production deployment — it creates real cloud resources, connects the repo, and exposes the app publicly). Recommend explicit user confirmation before executing.

Risks Identified

  • Central technical risk (also Level 3): the Express API + Agent SDK long-running loops may exceed Vercel serverless function timeouts. Interview generation runs bounded model calls; the bounded code scout (spec 5) is the longest-running operation in the whole tool and is the real concern. Mitigation — choose at execution against measured durations, options A/B/C:
    • A. Run the Agent SDK routes on Vercel with extended function duration / Fluid compute, if the limits cover the bounded operations.
    • B. Keep the SPA on Vercel but host the Express API + Agent SDK on a long-running Node host (Railway, Render, or Fly), sharing the same Neon database.
    • C. Offload the scout and any long generations to a background or queued worker so the request handlers stay short.
    Recommended: (A) for the interview, since its calls are short and bounded; escalate the scout to (C) or (B) if it exceeds Vercel's limits. The decision is finalized at execution against measured run times.
  • Neon serverless connection exhaustion. Mitigation: use the pooled connection string with SSL through the one central Knex pool (§10.6); never open a per-request connection.
  • Secret leakage into the client. Mitigation: keep every secret in Vercel env only; ship only VITE_ public config in the bundle (§17.3).
  • Migration drift between dev (Docker) and prod (Neon). Mitigation: run the identical Knex migrations against both environments, and run them against Neon before first traffic (§10.3).

Blast radius: the whole app plus all of the user's hosting accounts (Neon, Vercel, GitHub). Depends on all of specs 1 through 7 being executed first.

Pushback

  • The stated target is "Vercel + Neon," but if the bounded code scout cannot fit Vercel's function limits even with extended duration, option (B) — splitting the API onto a long-running host while the SPA stays on Vercel — is the pragmatic answer, not a failure. This is a real architectural fork; flag it for the user's call at execution rather than forcing the scout into a timeout.

Runtime Decision — Code Scout (resolved ahead of deploy)

Decision: The central runtime risk above (the scout exceeding Vercel's serverless function timeout) is resolved as Option C — offload the scout to a queued background worker, and it is already built and green as part of the code-scout plan (see plans/06242026-pm-tool-code-scout Rev 5). The scout no longer runs inline in a request.

How it runs (§21, deploy T3/T7)

  • Queue: a DB-backed scout_jobs table (migration 20260624000005_scout_jobs_schema.ts) — status (pending|running|done|failed), bounded attempts, last_error dead-letter detail; reached through an owner-verified session (§11.7). It applies to Neon via the same knex migrate:latest run as every other migration (T5) — no extra deploy step.
  • Enqueue (request path): POST /sessions/:id/scout enqueues a pending job and returns 202 in ~5ms — the handler is now trivially short and Vercel-compatible. GET /sessions/:id/scout returns {status, findings? when done} for polling.
  • Processor on Vercel: a guarded internal endpoint POST /internal/scout/process (shared-secret header, §5.4) is triggered by Vercel Cron every minute (see vercel.json crons) to drain the queue; the processor function gets a longer maxDuration (default 60s in vercel.json functions; raise toward the plan ceiling at T3/T7 against measured run times) while the request handlers stay short. The scout CodeScoutService is reused, not reimplemented (§21.3).
  • Processor locally: npm run scout:work runs the same processor in a poll loop — no cron needed in dev.
  • Secret: set SCOUT_WORKER_SECRET in Vercel env (server-side only, never in the bundle, §5.1/§17.3) and configure Vercel Cron to send the x-scout-worker-secret header; when unset the trigger is fail-closed.

Effect on T3/T7: the A/B/C fork is settled — interview generation stays on Vercel functions (Option A, short bounded calls); the scout is Option C (Cron-processed DB queue). T7 should record this as the chosen runtime; the rollback path is unchanged (revert the Vercel deployment; Neon and Docker share identical migrations, including scout_jobs). Verified: backend + frontend tsc 0, lint clean, check:conventions --strict 0 backend hard violations, 155 backend + 2 frontend tests green, and one live background run (enqueue ~5ms, scan ~15s out of band, findings cached).
