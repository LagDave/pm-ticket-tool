Why

The PM tool reads as a competent-but-generic dark SPA. The ask is a distinct identity: a refined serif + "futuristic" dark aesthetic with an animated WebGL backdrop, motion choreography, and a real light mode — so the product feels considered, not templated. impeccable detect also flagged two concrete tells to fix (see Context).

What

A full visual overhaul delivered behind the existing CSS-variable theming, with five additive layers and one cleanup:

  • Theme system — light + dark palettes as CSS variables, a persisted toggle (dark default), no hardcoded colors.
  • Typography & tokens — a more deliberate serif display voice with a futuristic mono/sans accent; refreshed spacing, radii, elevation.
  • WebGL backdrop — a single Three.js canvas mounted once at the app shell, behind every route, with hard perf/accessibility guards.
  • Motion — GSAP drives the backdrop + scroll choreography; Framer Motion stays the layer for React component transitions (a documented boundary, not two libs doing the same job).
  • Component restyle pass — dashboard, projects, project detail, wizard, modals, ticket views, in both themes.
  • Cleanup — fix the two impeccable detect findings and re-run it clean.

Done when: both themes pass contrast, the backdrop runs app-wide but degrades gracefully (reduced-motion + low-end + hidden-tab), tsc/lint are clean, impeccable detect is clean, and the initial bundle isn't blocked by Three.js.

Context

Relevant Files

  • frontend/src/index.css — the single stylesheet; all theming is via CSS custom properties in :root. The light/dark palettes and tokens live here.
  • frontend/src/App.tsx / app shell — where the persistent backdrop canvas and the ThemeProvider mount once (not per-route).
  • frontend/src/main.tsx — app entry; imports index.css.
  • frontend/src/components/ui/ThinkingLoader.tsx, Select.tsx — existing Framer Motion usage; the reference for component-level motion that stays on Framer.
  • Pages: Dashboard.tsx, ProjectsManager.tsx, ProjectDetail.tsx, InterviewWizard.tsx + interview/ticket/bits components — the restyle surface.

impeccable detect findings (to fix)

  • index.css:351 [side-tab] border-left: 2px solid var(--accent-deep) — the "AI UI" tell; replace with a subtler treatment.
  • index.css:561 [layout-transition] transition: width — animate transform instead to avoid layout thrash.

Patterns To Follow

  • Theme exclusively through CSS variables (existing convention) — components never hardcode hex.
  • One loader, one motion vocabulary — extend the existing tokens; do not fork parallel systems.
  • prefers-reduced-motion is already respected in index.css; every new animation must honor it.

Reference: the current index.css :root token block is the closest analog for how the new palettes/tokens should be structured.

Constraints

Must

  • Theme via CSS variables only; both light + dark defined for every token.
  • Lazy-load / code-split Three.js + GSAP so first paint is never blocked.
  • One shared backdrop canvas mounted at the shell — never remounted per route.
  • Backdrop guards: cap devicePixelRatio, cap FPS, pause on hidden tab, static fallback under prefers-reduced-motion and on low-end/no-WebGL.
  • Document the Framer (components) vs GSAP (backdrop/scroll) boundary in code comments.
  • Maintain WCAG AA contrast in both themes.

Must Not

  • No hardcoded colors in components.
  • No migrating existing Framer usage to GSAP (no churn on working motion).
  • No backend changes — this is frontend-only.
  • No blocking the main thread with the WebGL loop on low-end devices.

Out Of Scope

  • Backend, API, data model — untouched.
  • Resolving the dev-only npm-audit vulnerabilities introduced by impeccable (tracked separately).
  • Migrating the backdrop to @react-three/fiber — using raw three for the single canvas (revisit only if scene authoring gets complex).

Risk

Level: 3 (Structural — app-wide visual change, new heavy runtime deps, perf/accessibility surface).

Risks Identified

  • App-wide WebGL perf/battery/low-end. Mitigation: single shared canvas, DPR cap (≤1.5), FPS cap, pause via Page Visibility API, static fallback under reduced-motion / no-WebGL, lazy import so it never blocks paint.
  • Two animation libraries (Framer + GSAP) drift into a parallel abstraction. Mitigation: a hard, documented boundary — Framer = React component transitions, GSAP = backdrop + scroll timelines only. Reviewed at execution.
  • Light mode breaks contrast / reveals hardcoded colors. Mitigation: token audit; define every variable for both themes; contrast-check key surfaces.
  • Bundle growth (Three.js + GSAP ≈ 200KB+ gzip). Mitigation: code-split the backdrop + GSAP into an async chunk; tree-shake Three imports.
  • impeccable dev dependency carries 1 critical + 1 high + 3 moderate vulns. Mitigation: dev-only (never bundled); evaluate npm audit separately; consider running it via npx without keeping it as a dependency.

Blast radius: index.css (every screen), the app shell (App.tsx/entry), and every page + UI component. Frontend only — no backend consumers.

Pushback

  • GSAP alongside Framer is redundant by default. Recommended alternative was Framer + Three's own loop (no GSAP). Proceeding with GSAP per your call — scoped strictly to the backdrop/scroll so it doesn't overlap Framer.
  • App-wide WebGL is the costliest option. Recommended landing/dashboard-only. Proceeding app-wide per your call — mitigated by the single-canvas + guards above; if mid-device FPS suffers, fall back to the CSS/Canvas backdrop without changing the rest of the design.
