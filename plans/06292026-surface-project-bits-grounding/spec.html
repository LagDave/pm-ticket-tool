Why

The interview engine is wired to ground questions in a project's bits (project-context snippets), but in practice sessions keep running ungrounded — e.g. the scope flow asked "What is the current login method being replaced?" when PuzzlePulse's bits already settle auth. Two real gaps cause this: the re-run (clone) path silently drops the project link, and the project picker defaults to "No project" so a session is created ungrounded by accident. On top of that, even when grounding is present, the PM cannot see which bits informed this task — the grounding is invisible, so there is no way to trust or verify it. We fix the passing and make the related bits visible.

What

Three outcomes, verifiable in the running app:

  • Bits are passed reliably. A re-run of a grounded session stays grounded (cloneSession preserves project_id). The interview shows an explicit grounding-state badge so an accidentally-ungrounded session is obvious, and it distinguishes "no project attached" from "project attached but no active bits."
  • Related bits are shown. During the feature-scope step, a read-only panel lists the project bits related to this request — a model-selected subset — with a one-line "why," plus a "show all project context" expander revealing every active bit. Each grounded option also names the bit it came from inline.
  • Done = you can see it. Done is verified by watching a grounded session surface its related bits and an ungrounded session show the badge, against the running app (test.html, Layer 2).

Patterns To Follow

  • Backend (§6.1, §7.1, §7.4): all DB stays in models/ (ProjectBitModel.listActiveByProject); the service orchestrates; the agent owns prompt + structured output. Mirror the existing grounded path in interviewAgent.ts (buildBitsBlock / GROUNDED_OPTION_RULES).
  • Boundary validation (§11.2): the new relatedBits field is validated in generatedBatchSchema before it is ever persisted — never trust raw model output.
  • Frontend (§12.3, §14.3, §15.1): the panel is a presentational component in components/interview/; it reads server state from the useInterview hook — no fetch (§14.2), no business logic. Mirror the existing skipped-questions rendering in QuestionBatch.tsx.
  • No any (§17.2): every new field is typed in types/ on both sides.

Reference analogs: backend — the grounded path already in src/agents/interviewAgent.ts (the skipped field is the closest existing "model-emitted, schema-validated, persisted-in-batch, rendered-in-UI" feature — relatedBits mirrors it end to end). Frontend — the skipped-questions block in QuestionBatch.tsx.

Constraints

Must

  • Generation must still receive all active bits — the relatedBits subset is display only and must never change what is passed to suppression/grounding.
  • relatedBits is nullable; ungrounded batches and all pre-existing persisted turns resolve to "no related bits" without error (backward compatible).
  • A failed/absent relatedBits must never block question generation — degrade to the existing behavior.
  • Validate the new field at the boundary (§11.2); type every field, no any (§17.2).
  • The grounding-state badge must distinguish: no project · project with no active bits · grounded.

Must Not

  • No second/standalone LLM call for relevance — fold the field into the existing batch call (see Pushback).
  • No change to the project-picker default (decision: keep "No project", surface state instead).
  • No new dependencies. No DB queries outside models/ (§7.4). No fetch outside api/ (§14.2).
  • No drive-by refactors of the interview engine beyond these fields.

Out Of Scope

  • Ticket-draft "context used" surface — the ticket already has a codebase_grounding section (ticketAgent.ts) that maps the work to the project context. Not duplicated here.
  • Changing the project-picker default, or any bit-reconciliation / bit-authoring flow.
  • A schema migration — no new DB columns; relatedBits rides the existing batch JSONB, the grounding summary is computed on read.

Risk

Level: 2 (Concern). Mitigations below keep it from rising.

Risks Identified

  • The relevance subset can wrongly hide a relevant bit (model error). Mitigation: (a) generation still gets all bits, so suppression/grounding correctness is untouched; (b) the panel has a "show all project context" expander, so the subset is a highlight, never a gate.
  • Adding a model output field could push the model off-schema and fail batch generation. Mitigation: relatedBits is optional/nullable in generatedBatchSchema; on absence we persist null and the UI falls back. The boundary validator already rejects only truly malformed batches.
  • getState now loads the project + active bits on every state read (it is polled). Mitigation: active-bit sets are small (project context, typically <30 rows) and read through the indexed listActiveByProject (§10.4); only when project_id is set. Acceptable; revisit with caching only if profiling shows cost.
  • An option's groundingRef may name a bit not in the related subset. Mitigation: resolve groundingRef against the full active-bit dictionary in InterviewState.grounding; fall back to the raw key if unresolved.

Blast radius: parseBatchOrThrow + generatedBatchSchema (batch parsing) · existing persisted turns (must stay readable with no relatedBits) · getState / getSessionState consumers (added grounding field) · the re-run path (cloneSession → controller/route → frontend SessionList.onReRun) · QuestionBatch.tsx, OptionCard/OptionDeck rendering.

Pushback

  • You chose "add a relevance step" over "highlight referenced." A separate relevance LLM call adds latency, cost, and a new failure path on every session, for a result the model can already produce in-line. Recommended approach (taken in this spec): honor the intent — a model-selected related subset — by adding a relatedBits field to the existing batch call, and pair it with the "show all" expander so nothing the model omits is ever lost. This gives the precision of a relevance step with the safety of show-all, at zero extra round-trips. If you truly want a pre-question, session-level relevance pass independent of batch 1, say so and we add a task — but it is strictly more expensive for little gain.
