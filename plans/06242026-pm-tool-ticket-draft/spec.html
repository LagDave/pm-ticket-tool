Why

This produces the durable artifact — a consistent ticket built from the interview's decision_record. The ticket format is the real product value: every ticket comes out in the same shape (As a role, I want capability, So that benefit) with Given/When/Then acceptance criteria, an effort tier, and a short context summary.

The interview engine (spec 2) turns a vague request into structured decisions, but those decisions are not a deliverable on their own. A PM needs something they can read, refine, and hand off. This spec is the step that turns saved decisions into that hand-off artifact and lets the PM own it — edit it, comment on it, and mark it final.

What

A generator built on an Agent SDK structured-output call. It takes a session's decision_record plus the original_request and produces four fields: user_story (the As a / I want / So that line), acceptance_criteria (an array of Given/When/Then blocks), effort (a complexity tier, not hours), and a context summary. The result persists to foundation's tickets table. The PM can then edit the ticket inline, add comments, and mark it final — which flips status from draft to final and bumps version. The v1 output format is copy-paste Markdown.

Done when: a completed (or stopped) session yields a persisted, editable ticket in the standard format; inline edits and comments persist; finalize bumps the version; the PM can copy the ticket as Markdown; and npx tsc --noEmit plus npm run check:conventions --strict pass.

Constraints

Must

  • Match the story format: As a role, I want capability, So that benefit, plus Given/When/Then acceptance criteria.
  • Treat effort as a complexity tier (for example XS / S / M / L / XL), never a count of hours — LLM hour estimates are unreliable, so be honest about it.
  • Version the ticket (tickets.version); keep it editable and persisted; store comments in their own table.
  • Validate the structured-output payload at the boundary before persisting (§11.2); on a malformed shape, fail with a typed error (§8.3).
  • Read the Agent SDK key server-side only; the SPA never sees it (§5.1).

Must Not

  • Push to Jira, Azure DevOps, or GitHub Issues — that is a later, post-v1 feature.
  • Fabricate hour estimates or any numeric time prediction.
  • Put DB calls outside models/ (§7.4); add a second state library (§15.4); call fetch/axios outside the api client (§14.2).
  • Build the dashboard, the interview itself, or any tracker integration here.
  • Ship an mssql.sql migration — this app is Postgres-only.

Out Of Scope

  • Tracker push/sync (Jira/Azure/GitHub), the dashboard, the interview engine, and triage routing — each owned by its own spec.

Risk

Level: 2 (reads an upstream table, writes a foundation shell; no existing consumers in production yet).

Risks Identified

  • Malformed Gherkin / acceptance criteria from the model. Mitigation: request a strict structured-output schema (an array of { given, when, then } objects) and validate it at the boundary (§11.2) before persisting; reject and surface a typed TicketError (§8.3) on a bad shape rather than writing junk.
  • Effort overconfidence — the model sounds certain about complexity. Mitigation: constrain effort to a small enum of tiers and render a fixed "verify with engineering" note next to it; never expose or store hours.
  • Concurrent edits — two saves race and one silently overwrites the other. Mitigation: guard updates with the version field (optimistic concurrency) — an update carries the version it read; a mismatch is rejected so the PM re-reads before saving.
  • Re-running generation clobbers PM edits. Mitigation: generation writes the draft; once a PM has edited or finalized, re-generation is an explicit action that bumps version, not a silent overwrite. Flag the exact behavior during execution.

Blast radius: reads decision_record (produced by spec 2, interview-engine); writes tickets (the foundation shell) and the new ticket_comments table. Downstream consumers: the dashboard (spec 4) lists and opens these tickets, and triage's simple path (spec 7) reads the generated ticket. No code consumers exist yet — both are later specs.

Pushback

  • Comments storage — child table vs jsonb column. A jsonb array on tickets is fewer moving parts, but it is hard to query, paginate, or attribute per author, and every comment write rewrites the whole ticket row (a concurrency hazard against the version guard above). Recommended: a small child table ticket_comments (one row per comment, FK to tickets, indexed on ticket_id) — queryable, attributable, and it keeps comment writes off the ticket row. Flag your call; if you expect comments to stay tiny and never queried, the jsonb column is the cheaper path.
