Why

Every later feature (interview engine, ticket draft, dashboard, code scout, grounded options, triage) needs a platform to build on. This stands up that platform: the TypeScript + Express + Knex API, the React 19 + Vite SPA, a Dockerized PostgreSQL on host port 8765, the persisted interview/ticket schema, and the step-wizard shell with request entry.

Persistence is first-class from day one because the dashboard lets PMs leave, resume, finish, and re-run interviews — a resumable interview cannot live in memory (Phase-2 risk L3). Building foundation correctly now makes it the reference implementation the other six specs mirror, since this is a greenfield repo with no existing analog.

What

A booting full-stack app. docker compose up runs Postgres on host port 8765; the API connects through one central Knex connection and validates required config at startup. Base migrations create interview_sessions, interview_turns, decision_record, and tickets. A PM opens the wizard, types an initial request on step one, and that persists a session row retrievable by id.

Done when: the stack boots, the DB connects, creating a session from the request-entry screen writes a row that GET /sessions/:id returns, and npm run check:conventions --strict plus npx tsc --noEmit pass on both trees. No AI, no scout, no ticket generation, no dashboard browsing yet.

  • Backend layering Routes → Controllers → Services → Models (§7.1); thin routes (§7.2); controllers never touch the DB (§7.3); all queries in models/ (§7.4 🔎).
  • Response envelope { success, data, error } (§8.1) via shared builders (§8.2) + typed InterviewError with one status mapper (§8.3).
  • Frontend triad Component → hook → api/ client (§14.1); the client is the only fetch path (§14.2 🔎); server state is React Query (§15.1).
  • Pino, no console.* (§9.1 🔎 / §17.1 🔎); no any (§17.2 🔎); secrets server-side, only VITE_ in the bundle (§5.1, §17.3).

Reference analog: backend mirrors src/controllers/gbp-automation/ (§6.1, Constitution Part VI); frontend mirrors the frontend/src/api/ + React Query triad (§12.1). Constraints

Must

  • Conform to the Constitution; cite §N.M on any deviation.
  • Single Knex connection (§10.6); schema only via migrations (§10.3); index session_id / owner / status (§10.4).
  • Validate config at startup, fail fast (§5.6); secrets via env only (§5.1).
  • Owner-scope every session/ticket query in models/ (§11.7, §5.5).
  • Write-through persistence: schema must let a later engine replay state for resume.

Must Not

  • Implement interview/AI logic — that is the interview-engine spec.
  • Implement the code scout or any repo provider.
  • Build dashboard browsing / resume UI — that is the dashboard spec.
  • Put DB calls outside models/ (§7.4); add a second state library (§15.4).
  • Ship an mssql.sql migration — this app is Postgres-only (see Risk).

Out Of Scope

  • Question generation, ticket drafting, GitHub/Azure providers, dashboard, triage routing, push-to-tracker.

Risk

Level: 2 (greenfield scaffolding; no existing consumers).

Risks Identified

  • Resume/re-run needs durable interview state (L3 principle). Mitigation: design the schema here — interview_turns as a write-through log and decision_record as structured rows (not chat text) — so "resume" is replaying saved state; the engine spec consumes it.
  • Migration convention mandates mssql + pgsql + knex. Mitigation: this app standardizes on Postgres (Docker :8765), so we ship knexmigration.js + pgsql.sql and skip mssql.sql. Flag if SQL Server parity is needed.
  • No reference analog to mirror (greenfield). Mitigation: foundation IS the analog — build it exactly to §6.1 / §12.1 so 2–7 cite "mirror foundation."
  • Secrets (ANTHROPIC_API_KEY now, repo tokens later). Mitigation: env-only (§5.1), server-side only, never VITE_ (§17.3).

Blast radius: none — new project, no existing consumers.

Pushback

  • Tenant scope (§5.5 / §11.7). This is an internal PM tool, so v1 scopes by owner_user_id rather than full org multi-tenancy; an organization_id column is included (nullable) to keep the door open. Recommended alternative if you expect multiple client orgs soon: make organization_id required and scope by it now. Flag your call.

Rev 3 - 2026-06-24

Change: User QA: auto-generated session title + no-em-dash output rule. (1) New migration 20260624000006_session_title_schema.ts adds a nullable title (text) column to interview_sessions (round-trips clean: apply, rollback drops it, re-apply restores). (2) New titleAgent (src/agents/titleAgent.ts) mirrors the existing agent pattern — LOW reasoning effort, claude-opus-4-8 primary / claude-sonnet-4-6 fallback via config.TITLE_GENERATION — generating a concise title (max ~8 words, sentence case, no surrounding quotes, no em-dashes) from either a request string or a finalized ticket; output is sanitized structurally (sanitizeTitle) so quotes/dashes/over-length never persist. (3) New TitleService wraps the agent with model fallback + boundary re-validation, degrading to null on any failure (a title is never a hard gate). (4) Session create (InterviewSessionService.createSession and cloneSession) generates + persists the title from original_request at the first prompt. (5) title exposed on session API responses (create/get/list flow through the row) and added to the session shape in src/types/interview.ts (string | null) and frontend/src/types/interview.ts (title?: string | null). (6) InterviewSessionModel.updateTitleForOwner (owner-scoped, §11.7) persists the title; create persists it on insert.

Reason — User QA: The dashboard showed the raw request as each session's label; a concise generated title is generated twice — once from the request at create (immediate display title) and once refined from the finalized ticket, replacing the stored title. The no-em-dash rule ("Never use em-dashes (—) or en-dashes (–) in any output. Use commas, periods, parentheses, or hyphens instead.") was added to every generation prompt (titleAgent, interviewAgent grounded + ungrounded via shared BASE_RULES, ticketAgent, triageAgent), and em-dashes were scrubbed from user-facing backend strings (the ticket Markdown effort note in TicketMarkdownService; the GitHub auth error in GitHubCodeContextProvider). Title regeneration after finalize lives in the ticket-draft spec (Rev 2 there).
